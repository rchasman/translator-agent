# translator-agent

$10,000-ის ლოკალიზაციის პრობლემა, გადაწყვეტილი 90 წამში.

კომპანიები სააგენტოებს უხდიან $0.10–0.25 სიტყვაზე თავიანთი საიტების ლოკალიზაციისთვის. 5,000-სიტყვიანი საიტის 10 ენაზე თარგმნა ღირს $5,000–12,000 და სჭირდება 2–4 კვირა. ყოველ ჯერზე, როცა სათაურს ცვლით, ლოკმა თავიდან იწყება.

ეს ინსტრუმენტი ამას ერთი ბრძანებით აკეთებს, 71 ენაზე, თქვენი შექმნის პროცესში:

```bash
bunx translator-agent -s ./dist -l all
```

არანაირი სააგენტო. არანაირი ელცხრილი. არანაირი მომწოდებლის ბმა. არანაირი რეგისტრაცია. თქვენი გასაღებები, თქვენი შექმნა, თქვენი ენები.

## რატომ მუშაობს ეს

თარგმანი გადაწყვეტილი პრობლემაა. ლოკალიზაცია კი არა.

Google Translate-ს შეუძლია "ჩვენი ბოზები მუშაობენ ამაზე" იაპონურად გადაკეთება. რაც მას არ შეუძლია — ამოიცნოს, რომ ხუმრობა იაპონიაში არ შეედება და ჩაანაცვლოს რაღაცით, რაც მუშაობს — მაგალითად, ინჟინერთა გუნდზე მინიშნება, რომელიც ღამეს მუშაობს, რაც კულტურულად მისაღები და კონტექსტში სახალისოცაა.

ეს ინსტრუმენტი არ თარგმნის. იგი **ტრანსკრიერებს** — იგივე პროცესი, რისთვისაც სარეკლამო სააგენტოები $50,000-ს იხდიან კამპანიის ბაზრებზე ადაპტაციისთვის. ოღონდ LLM-მა უკვე იცის ყველა კულტურა, ყველა ფრაზეოლოგიზმი, ყველა ფორმატირების წესი. მან იცის, რომ:

- `$49/month` იაპონიაში ხდება `月額6,980円` — არა "$49" იენის სიმბოლოთი მიკებული
- სარკაზმი ინგლისურში ხოცავს, ოფიციალურ იაპონურში კი კვდება
- "ქაღალდებში ჩაძირვა" ფრანგულში ხდება "noyade administrative" — ნამდვილი ფრანგული გამოთქმა, არა სიტყვა-სიტყვით თარგმანი
- გერმანელები ბოზების ხუმრობას ტოვებენ, რადგან Hamsterrad (ბოზის ბორბალი) ნამდვილი გერმანული ფრაზეოლოგიზმია
- ბრაზილიელებს არაოფიციალური რეგისტრი უნდა, თუ არადა ისე ჟღერს, თითქოს რობოტმა დაწერა

მოდელი კლასიფიცირებს ყოველ სტრიქონს. UI ლეიბლები პირდაპირ თარგმანას იღებენ. მარკეტინგული ტექსტი ადაპტირდება. ხუმრობა მთლიანად იქმნება სამიზნე კულტურისთვის.

## რა ხდება გაშვებისას

მიუთითეთ თქვენი შექმნის შედეგზე. იგი კლონავს მთელ ფაილთა ხეს ლოკალზე — თარგმნის ტექსტურ ფაილებს, კოპირებს სტატიკურ აქტივებს და აგენერირებს განლაგებისთვის საჭირო ყველაფერს:

```
your-site/                          translations/
  index.html                          middleware.ts        ← ლოკალის დადგენა
  about.html             →            _locales.css         ← ტიპოგრაფია სკრიპტის მიხედვით
  css/style.css                       ja/
  js/app.js                             index.html         ← lang="ja", ტრანსკრიირებული
  images/logo.png                       about.html
                                        _locale.css        ← Noto Sans JP, 1.8 სტრიქონთა სიმაღლე
                                        css/style.css      ← კოპირებული
                                        js/app.js          ← კოპირებული
                                        images/logo.png    ← კოპირებული
                                      ar/
                                        index.html         ← lang="ar" dir="rtl"
                                        _locale.css        ← Noto Naskh Arabic, 110% შრიფტი
                                        ...
                                      de/
                                        ...
```

ყოველ HTML ფაილში იდება `lang` და `dir="rtl"`. ყოველ ლოკალს უვლის CSS სწორი ფონტური სტეკით, სტრიქონთა სიმაღლით და ტექსტის მიმართულებით. იქმნება Vercel middleware, რომელიც კითხულობს `Accept-Language`-ს და გადამისამართებს სწორ ლოკალზე.

განალაგეთ Vercel-ზე. ტოკიოს მომხმარებელი ხედავს იაპონურს Hiragino Sans-ით 1.8 სტრიქონთა სიმაღლეზე. კაიროს მომხმარებელი ხედავს RTL არაბულს Noto Naskh-ით 110% ზომაზე. ბანკოკის მომხმარებელი ხედავს ტაიურს `word-break: keep-all`-ით, რადგან ტაიურს ადგილები არ აქვს. არანაირი კონფიგურაცია.

## მოიტანეთ თქვენი გასაღებები ან არ მოიტანოთ

```bash
# თქვენ გაქვთ გასაღებები — ლოკალურად მუშაობს, LLM პროვაიდერს პირდაპირ უხდით
export ANTHROPIC_API_KEY=sk-...
bunx translator-agent -s ./dist -l all

# არ გაქვთ გასაღებები — უბრალოდ მუშაობს
# ავტომატურად იყენებს ჰოსტირებული სერვისს
# იხადეთ თარგმანზე USDC-თი x402-ით — არანაირი რეგისტრაცია, არანაირი ანგარიში
bunx translator-agent -s ./dist -l all
```

იგივე ბრძანება. თუ API გასაღებები არსებობს, ლოკალურად მუშაობს თქვენს პროვაიდერთან. თუ არადა, ჰოსტირებულ API-ს წვდება და იხდის მოთხოვნაზე [x402](https://x402.org)-ით — HTTP 402 გადახდის პროტოკოლით. თქვენი კლიენტი USDC-ს იხდის Base-ზე, მიიღებს თარგმანებს. არანაირი ავტორიზაცია, არანაირი მომწოდებლის ურთიერთობა, არანაირი ინვოისები.

მხარს უჭერს Anthropic-სა და OpenAI-ს. მოიტანეთ რომელიც მოდელი გნებავთ:

```bash
bunx translator-agent -s ./dist -l all -p openai -m gpt-4o
```

## კეშირება

თარგმანები შექმნის პროდუქტებია. გენერაცია შექმნის დროს, გამოსული კეშირება, გამოტოვება წყარო არ რომ შეცვალა.

### Vercel

Vercel ავტომატურად კეშირებს შექმნის შედეგს. დაამატეთ `postbuild` და მზადაა:

```json
{
  "scripts": {
    "build": "next build",
    "postbuild": "bunx translator-agent -s ./public -l all -o ./public"
  }
}
```

### CI

```yaml
- uses: actions/cache@v4
  id: translations
  with:
    path: translations/
    key: translations-${{ hashFiles('src/messages/**', 'public/**/*.html') }}

- if: steps.translations.outputs.cache-hit != 'true'
  run: bunx translator-agent -s ./src/messages -l all
  env:
    ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

წყარო უცვლელი = კეშის დარტყმა = ნული LLM გამოძახება = ნული ღირებულება.

## პარამეტრები

```
გამოყენება: translator-agent [პარამეტრები]

პარამეტრები:
  -s, --source <path>      სკანირებისთვის საწყისი დირექტორია ან ფაილი
  -l, --locales <locales>  სამიზნე ლოკალები, მძიმით გაყოფილი ან "all" 71 ენისთვის
  -o, --output <path>      გამოსავალი დირექტორია (ნაგულისხმევი: "./translations")
  -p, --provider <name>    anthropic | openai (ნაგულისხმევი: "anthropic")
  -m, --model <id>         მოდელის გადაფარვა
  -c, --concurrency <n>    მაქსიმალური პარალელური LLM გამოძახებები (ნაგულისხმევი: 10)
  --api-url <url>          ჰოსტირებული სერვისის URL (ავტო-გამოყენება API გასაღებების გარეშე)
```

| გაფართოება | სტრატეგია |
|---|---|
| `.json` | მნიშვნელობების თარგმანი, გასაღებების შენარჩუნება |
| `.md` / `.mdx` | ტექსტის თარგმანი, სინტაქსის შენარჩუნება |
| `.html` / `.htm` | ტექსტის თარგმანი, თეგების შენარჩუნება, `lang`/`dir`-ის ჩადება |
| ყველაფერი სხვა | ყოველ ლოკალის დირექტორიაში კოპირება |
